## 共用工具函數設計細節

### 1. 目標

把「Judge0 payload 建立 / 提交」和「Judge0 結果輪詢解讀」抽成 shared helper，讓：

- `app_oj/views.py` 和 `app_management/views.py` 都能重用相同邏輯
- 兩個 view 只保留各自的權限、DB side effect、流程差異
- 會減少重複、也降低維護成本

---

## 2. 建議放置位置

建立一個新檔案：

- `oj-web/app_oj/judger_utils.py`

此檔案負責：
- source code 處理
- judge payload 構造
- Judge0 提交
- Judge0 結果查詢
- 結果解讀

然後：
- `app_oj/views.py` import 這些 helper
- `app_management/views.py` 也 import 這些 helper

這樣可以避免目前 `app_management/views.py` 反向依賴 `app_oj/views.py` 的混亂。

---

## 3. 具體 helper 函數

### 3.1 source code / payload helper

在 `judger_utils.py` 建：

- `prepare_source_code_for_judge(problem, source_code)`
  - 如果題目有 template，先把 `prepend/template/append` 拼起來
  - 最後回傳 base64 編碼的 source_code

- `load_additional_files_for_database_problem(problem)`
  - 若 `problem.categories` 有「資料庫」字樣，讀 `judger_env/sqlite_env.zip`
  - 回傳 `encoded_additional_files` 或 `None`

### 3.2 Judge0 提交 helper

- `submit_to_judge(source_code_b64, problem, test_cases, additional_files=None)`
  - 建 `judge0_payload`
  - 對每個 `test_case` 呼叫 `judge(...)`
  - 回傳 token list

或更細分：

- `build_judge_payload(...)`
- `submit_judge_task(payload)`：只負責 POST Judge0

### 3.3 Judge0 結果查詢 helper

- `fetch_judge_result(task_token)`
  - GET `judger_url/{token}?base64_encoded=true`
  - 回傳原始 judge_result dict

- `interpret_judge_result(judge_result)`
  - 根據 `status.id` 轉成統一輸出：
    - `result_status`
    - `judge_status`
    - `judge_compile_output`
    - `judge_status_description`
    - `is_done` / `needs_polling`
  - 這個 helper 將語意標準化，讓前端判斷一致

- `poll_judge_tokens(task_tokens)`
  - 依序查詢 tokens
  - 一個 token 若還在 `IN_QUEUE` 或 `PROCESSING`，就回 `JudgerInQueueOrProcessing`
  - 若遇第一個非 AC 結果，就回該錯誤
  - 全部 AC 則回 `JudgerSUCCESS`

## 4. 最合理的 refactor 方法

### `app_oj/views.py` 保留

- `submit_to_judger(request)`
  - 仍然做：
    - `contest_id` / `problem_id` 讀取
    - 比賽狀態檢查
    - `Submission` 建立 / 更新
    - AC 後禁止再送的判斷
  - 但 `source_code` 處理和 Judge0 提交改呼叫 shared helper

- `get_submission_result(request)`
  - 仍然做：
    - token 讀取
    - `poll_judge_tokens(...)`
    - DB 更新 `Submission`
    - `ContestRank` 更新
  - 但 Judge0 查詢和結果解讀改呼叫 shared helper

### `app_management/views.py` 保留

- `submit_to_judger(request)`
  - 仍然做：
    - 只針對 `Problem`，沒有 contest / submission DB
  - 但 source/template+DB 關聯的附檔處理改成 shared helper

- `get_manager_submission_result(request)`
  - 仍然做：
    - token 讀取
    - 呼叫同一套 `poll_judge_tokens(...)`
    - 回傳結果給前端
  - 不做任何 Submission / ContestRank 更新

## 5. 建議的 shared helper 模式

`judger_utils.py` 內部可設計成三層：

1. `prepare_*`：
   - `prepare_source_code_for_judge()`
   - `load_additional_files_for_database_problem()`

2. `submit_*`：
   - `submit_problem_to_judge(...)`

3. `result_*`：
   - `fetch_judge_result(...)`
   - `interpret_judge_result(...)`
   - `poll_judge_tokens(...)`

這樣上層 view 只做兩件事：

- `submit_to_judger`：準備好參數後呼叫 shared submit helper
- `get_submission_result` / `get_manager_submission_result`：呼叫 shared result helper，再補自己的 DB side effect

---

## 6. 為什麼這樣最合理

- 兩個 view 邏輯「真的不是完全一樣」：一個要寫 submission 與排名，另一個只做 admin 測試
- 但「Judge0 payload 與結果解讀」核心邏輯是同一份
- 因此最合理做法是：
  - `共用 judge 工具`
  - `保留兩個 view wrapper`
